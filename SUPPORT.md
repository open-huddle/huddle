# Getting support

Thanks for using Open Huddle. This page explains where to go for help, depending on what you need.

## Questions and discussion

For usage questions, deployment help, design discussion, or sharing what you are building, use **[GitHub Discussions](https://github.com/open-huddle/huddle/discussions)**. Please search existing discussions before opening a new one.

## Bug reports

If you believe you have found a defect, open a **[Bug report](https://github.com/open-huddle/huddle/issues/new/choose)** issue and fill in every section of the template. Bugs without reproduction steps are difficult to act on and may be closed.

## Feature requests

Open a **[Feature request](https://github.com/open-huddle/huddle/issues/new/choose)** issue *before* writing code. See [CONTRIBUTING.md](CONTRIBUTING.md) for why.

## Security vulnerabilities

**Do not use GitHub issues or Discussions for security problems.** Follow the responsible disclosure process in [SECURITY.md](SECURITY.md).

## Code of Conduct concerns

Report conduct issues privately to [conduct@open-huddle.org](mailto:conduct@open-huddle.org). See [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) for the enforcement process.

## Commercial support

Open Huddle is a community-driven project. Paid support, hosting, and professional services are not offered by the project itself. A list of third-party providers may appear here once the ecosystem develops.

## Response expectations

Open Huddle is a volunteer-maintained open-source project. Maintainers will respond to issues and discussions as time permits — typically within a week for bugs, longer for feature requests. Security reports follow the SLA documented in [SECURITY.md](SECURITY.md).
